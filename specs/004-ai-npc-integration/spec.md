# Feature Specification: AI NPC Integration — First Crossover

**Feature Branch**: `004-ai-npc-integration`
**Created**: 2026-04-26
**Status**: Draft
**Input**: First integration of the `ai/` backend with the `playtest/` Phaser frontend using Girl_1 sprite assets and the existing conversation/quest notification API.

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Approach NPC and Start a Conversation (Priority: P1)

The player walks up to the first AI-powered NPC (Girl_1) in the overworld. An interaction prompt appears near the NPC. The player triggers it and a conversation UI opens, connected to the `ai/` backend. The selected character archetype is driven by the archetype definitions in `ai/docs/prompts/`.

**Why this priority**: This is the single foundational slice — if the NPC can be approached and a conversation routed through the backend, the feature has a viable MVP.

**Independent Test**: Open the playtest, walk to the NPC sprite, press the interact key. A conversation panel should open and the first NPC greeting should appear as text from the AI backend.

**Acceptance Scenarios**:

1. **Given** the player is in the overworld, **When** they move within interaction range of the Girl_1 NPC, **Then** a subtle interaction prompt appears above or near the NPC.
2. **Given** the interaction prompt is visible, **When** the player presses the interact key, **Then** a conversation panel opens and an opening line from the AI character is displayed.
3. **Given** the conversation panel is open, **When** the player submits a text input, **Then** the NPC responds with a reply generated by the `ai/` backend.
4. **Given** the backend is unavailable, **When** the player attempts to speak to the NPC, **Then** the UI shows a graceful error and the panel does not break the game.

---

### User Story 2 — Receive a Quest Notification In-Game (Priority: P2)

After the NPC offers a quest (as determined by the `ai/` conversation lifecycle), the game receives and displays a small, non-intrusive in-game notification. The notification must not interrupt gameplay or occupy significant screen space.

**Why this priority**: The notification pipeline (`ai/src/notify/game-api.ts`) already exists and posts to `localhost:3000/quest/start` and `/quest/complete`. Wiring the game to consume these events is the second critical integration slice.

**Independent Test**: Manually trigger a quest-start POST to the local game API endpoint. A small toast-style notification should appear in the game without blocking the player.

**Acceptance Scenarios**:

1. **Given** a quest has been offered by the NPC, **When** the `quest_start` payload is received by the game, **Then** a small notification appears briefly in the HUD, showing a quest title only.
2. **Given** the player completes a quest, **When** the `quest_complete` payload is received, **Then** a small completion notification appears and disappears on its own.
3. **Given** a notification appears, **Then** the player can still move, open inventory, and interact normally — the notification does not block any action.

---

### User Story 3 — Character Archetype Is Selectable (Priority: P3)

Each NPC archetype defined in `ai/docs/prompts/` (enabler, opportunist, honest, mirror, parasite, general) can be selected or assigned to the NPC, but only one character asset (Girl_1) is used visually for this first pass.

**Why this priority**: The constraint says "only one character asset but allow for selection of all character archetypes." This story ensures the architecture supports all archetypes even though the visual is locked to Girl_1.

**Independent Test**: Change the assigned archetype in configuration or via a developer interaction and confirm the NPC responds with personality consistent with the chosen archetype while the sprite remains Girl_1.

**Acceptance Scenarios**:

1. **Given** an archetype is configured for the NPC, **When** the player speaks to the NPC, **Then** the conversation tone and quest behavior match that archetype's system prompt.
2. **Given** any archetype is selected, **Then** the NPC always renders using the Girl_1 sprite and animations.

---

### Edge Cases

- What happens when the AI backend (`localhost:1234`) is not running when the player opens the conversation?
- What happens if the quest notification server (`localhost:3000`) is unreachable — are notifications queued and retried (as the existing `pending-notifications.json` mechanism handles)?
- What if the player closes the conversation panel mid-response while text is streaming?
- What if the player rapidly reopens the conversation panel — does a second session open or is the existing one resumed?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The overworld MUST display the Girl_1 NPC sprite at a designated tile position with an idle animation.
- **FR-002**: The Girl_1 sprite sheet assets from `assets/_source/craftpix-net-242415-free-schoolgirls-anime-character-pixel-sprite-pack/Girl_1/` MUST be copied into the playtest asset pipeline and registered for use in the overworld scene.
- **FR-003**: The player MUST see a proximity interaction prompt when within range of the NPC.
- **FR-004**: Activating the interaction MUST open a conversation panel styled using pieces from `assets/ui/inventory/` (frames and tiles) as a consistent UI language.
- **FR-005**: The conversation panel MUST send player input to the `ai/` backend via its HTTP API and display the NPC reply.
- **FR-006**: The game MUST expose a local HTTP endpoint that accepts `quest_start` and `quest_complete` POST notifications from the `ai/` backend, consistent with the existing `notifyQuestStart` / `notifyQuestComplete` contract in `ai/src/notify/game-api.ts`.
- **FR-007**: Quest notifications received by the game MUST render as a small, auto-dismissing HUD element — no larger than a brief toast — and MUST NOT block player interaction.
- **FR-008**: The assigned NPC archetype MUST be configurable (e.g., via a data file or scene configuration) and MUST select the corresponding system prompt from `ai/docs/prompts/`.
- **FR-009**: Only the Girl_1 sprite is used for the NPC visual in this feature pass; the architecture MUST allow future substitution of other sprite sets per archetype without redesign.
- **FR-010**: The feature MUST be validated in-browser via Chromium, covering the conversation flow and quest notification display.

### Key Entities

- **NPC**: A Phaser sprite in the overworld with a designated position, archetype assignment, and interaction zone. Rendered exclusively as Girl_1 in this pass.
- **Conversation Panel**: An in-game UI overlay that presents the NPC dialogue, accepts player text input, and communicates with the `ai/` HTTP backend.
- **Archetype**: One of the six character personalities defined in `ai/docs/prompts/` (enabler, opportunist, honest, mirror, parasite, general). Determines the NPC's system prompt and behavioral logic.
- **Quest Notification**: A transient HUD element triggered by quest lifecycle events from the `ai/` backend. Contains a brief label only.
- **Game API Endpoint**: A lightweight local server (or in-page handler) that receives `quest_start` and `quest_complete` POSTs and dispatches them as in-game events.

---

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A player can go from entering the overworld to receiving a first NPC response within 10 seconds of triggering the interaction, under normal local conditions.
- **SC-002**: Quest notifications appear and auto-dismiss without the player taking any action, occupying no more than 10% of the screen's vertical height.
- **SC-003**: All six archetypes produce distinct conversation tones verifiable in a single play session by switching archetype config and restarting.
- **SC-004**: The conversation panel opens and closes without causing Phaser scene errors or leaving orphaned game objects (verified in browser devtools console).
- **SC-005**: The Girl_1 sprite displays its idle animation correctly at all standard playtest resolutions (verified via Chromium).

---

## Assumptions

- The `ai/` backend must be running locally (LM Studio on `localhost:1234`) during playtesting; this is assumed as a known developer setup requirement, not something the game auto-starts.
- A minimal local HTTP server (or in-page `fetch` handler on a known port, e.g., `localhost:3000`) will be introduced to receive quest notification POSTs — this matches the existing `GAME_API_URL` environment variable convention in `ai/src/notify/game-api.ts`.
- Conversation input is text-only for this first pass; voice or other modalities are out of scope.
- The Girl_1 sprite sheets are pixel-art PNG files compatible with Phaser's sprite/animation system.
- Mobile and touchscreen support is out of scope for this feature pass.
- The existing `pending-notifications.json` retry mechanism in the `ai/` backend handles cases where the game API is temporarily unavailable — no additional retry logic is required in the game.
- The conversation panel does not persist session history across page reloads in this first pass.
